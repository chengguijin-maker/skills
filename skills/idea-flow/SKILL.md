# Idea Flow

## 目标

把零散 idea 迅速收敛成一个**可执行、可验证、可迭代**的最小工作流。

融合两种核心能力：
- **项目基建**：单文件主控 WORK.md、AGENTS.md 约束入口、最小仓库结构、材料生命周期、git/worktree 规范
- **思维方法论**：问题定锚 → 规格收口 → 任务塑形 → 执行编排（按需）的四阶段收敛

默认采用**单文件优先**模式：
- 仓库根目录只保留一个主控文件 `WORK.md`
- 仓库根目录使用 `AGENTS.md` 作为 agent 约束入口
- 只有在内容过长、证据过重、生命周期独立时才拆文件

除非用户另行要求，不要引导到重型阶段文档流程；默认保持轻量闭环。

## 交互原则

这个工作流**不是静默执行**——它通过苏格拉底式提问帮助用户澄清自己的想法，在关键决策点呈现选项并等待用户拍板。

### 何时必须询问用户

以下场景必须停下来与用户交互，不能自行决定：

- **问题定义**：用户说的到底是问题还是方案？真正的痛点是什么？
- **边界划定**：V1 包含什么、不包含什么
- **方案分歧**：存在 2 个以上合理路径时（技术选型、架构模式、交互范式）
- **高代价决策**：数据模型、权限模型、同步/异步等难以后改的选择
- **范围变更**：发现需求比预期大，或出现新约束
- **假设验证**：某个假设的成立与否会显著改变方案走向

### 何时可以自行推进

以下场景做保守假设即可，不需要打断用户：

- UID 编号、文件命名等机械性操作
- 明显的非目标识别
- 低影响的实现细节（文案措辞、非关键 UI 微调）
- 模板填充和格式整理

### 苏格拉底式提问链

每次到达交互检查点时，按这个链条组织提问：

1. **回溯 + 假设陈述**：先总结已确认的事实，然后亮出你当前的假设——"根据你提到的 X，我假设 Y。如果不是，方案会完全不同。"假设让用户知道你在想什么，也让他们更容易纠偏。
2. **瞄准最弱维度**：在三个清晰度维度中找到最模糊的那个来提问：
   - **目标清晰度**：到底在解决谁的什么问题？
   - **约束清晰度**：边界、限制、不可变条件是什么？
   - **成功标准清晰度**：做成什么样算成功？

   下一问始终针对最弱维度。如果三个维度都同样模糊，优先问目标。
3. **呈现选项 + 蕴含链**：给出 2~4 个选项，每个选项必须包含：
   - 选项名称（一句话概括）
   - 你的假设（为什么你认为这个可能是答案）
   - 蕴含链（选了之后，下游会怎样——技术选型、V1 范围、优先级会如何变化）
   - 适用场景（什么情况下这个选项最优）
4. **推荐 + 路线预告**：标明推荐项并解释理由。然后预告下一步——"回答这个之后，我接下来会问 Z，因为..."让用户看到对话的方向感。
5. **等待用户决定**：不要替用户选择。

### 回应链（用户回答后）

用户做出选择后，不要直接跳到下一个不相关的问题。按这个链条处理回答：

1. **确认 + 追溯蕴含**：重述用户的选择，然后 trace implications——"你选了 A，这意味着我们的方向是 X，V1 的重点应该放在 Y 而不是 Z。"
2. **更新工作假设**：把新信息融入已知事实——"现在确认了：目标是…、用户画像是…、还不确定的是…"
3. **沉淀调研与决策**：检查本轮交互中是否产生了新的事实性调研（技术可行性验证、API 能力确认、竞品对比等）或路线选择。如果有：
   - 调研结论 → 登记到 WORK.md 调研登记表（RES 行），写明问题、结论、证据来源、重查条件
   - 路线选择 → 追加到决策链；如果属于高代价决策（数据模型、权限、同步/异步等难以后改的），同时写入 ADR-lite
   - 不要等阶段结束再批量补——**每轮交互产生的调研和决策当轮沉淀**，避免 session 断开或上下文压缩后丢失
4. **识别最弱维度 → 提下一问**：基于更新后的假设，找到最需要澄清的维度，按上面的提问链组织下一问。

这样每一轮对话都在**收窄不确定性**，而不是机械地走清单。用户应该能感受到：每个问题都是上一个回答的自然延伸，而不是无关的新话题。

### 决策链记录

对话中的决策会随 session 切换或上下文压缩丢失。每轮交互后，把关键决策写入 WORK.md 的**实施记录**区块，格式：

```
| 轮次 | 时间 | 问题 | 用户选择 | 蕴含 | 假设更新 |
|------|------|------|----------|------|----------|
| R1   | 03-16 14:30 | 核心痛点是什么？ | A: 维护成本高 | V1 先迁移变更热区 | 确认痛点=维护，非效率 |
| R2   | 03-16 14:35 | 能否渐进迁移？ | B: 必须保持线上可用 | 排除推倒重写 | 迁移=渐进式 |
```

这不是重型 ADR——只是一行一决策的时间线。好处：
- **防重复**：下一轮提问前先扫决策链，不问已决定的事
- **防矛盾**：新假设和已有决策冲突时必须显式处理
- **可恢复**：session 断了或上下文被压缩，从 WORK.md 恢复进度

### 决策链与 ADR-lite 的关系

决策链（实施记录中一行一决策）和 ADR-lite（§10 结构化记录）是两层不同粒度的沉淀：

- **决策链**：记录所有用户选择，包括低代价的方向确认。轻量、快速、每轮交互都写。
- **ADR-lite**：只记录高代价、难以后改的决策。结构化，含备选方案、取舍、证据。

**升级规则**：当决策链中某条选择满足以下任一条件时，必须同步写入 ADR-lite：
- 涉及数据模型、权限模型、同步/异步模式等架构级选择
- 涉及外部服务或供应商绑定
- 回退成本高（改了之后要动多个模块或需要数据迁移）
- 影响后续多个任务的依赖链

写入 ADR-lite 时，在决策链对应行的"备注"中标注 `→ ADR-XXX`，在 ADR-lite 行中也标注来源轮次 `R-N`，形成双向引用。

在阶段 0（项目基建）完成后就开始记录。每次用户做出选择都追加一行。阶段结束的标准动作总结中也包含决策链的快照。

### 问题路线图

在第一轮交互时，简短预告整体问题方向（2~3 句话），让用户知道大概会经历什么：

> "我会问 3~5 个问题来收敛方案：先锁定核心痛点 → 然后确认目标用户和使用场景 → 最后画 V1 边界。每个问题我会说明为什么现在需要回答它。"

这不是固定脚本——路线会根据用户的回答动态调整。但给出方向感比直接开问好得多。

### 提问格式

- **一次只问一个问题**——不要在同一条消息里塞多个问题。如果一个主题需要深入，拆成多轮对话。
- **优先多选题**——能用选项呈现的就用选项，减少用户认知负担。使用 AskUserQuestion 工具时提供 2~4 个选项，每个选项附带 description 说明取舍。开放式问题仅在选项无法覆盖时使用。
- 如果用户的回答暴露了新的未知项，继续追问而不是假装没看见。
- 用户明确说"你决定"或"帮我选"时，可以代为决定，但要写明理由。

### 响应简洁性

- **不要解释 idea-flow 是什么**——用户已经主动触发了这个 skill，不需要介绍工作流。直接进入正题。
- **不要用"好的，我来用 idea-flow 帮你..."之类的开场白**——直接开始回溯事实或提问。
- 每条响应聚焦在一件事上：回溯+假设 → 提出一个问题 → 等待。不要在提问中夹带方案雏形。

## 默认仓库结构

优先生成或维护这个最小结构：

```text
repo/
├── README.md
├── AGENTS.md
├── WORK.md
├── archive/
├── scratch/
├── src/
└── tests/
```

含义：
- `README.md`：一句话入口，指向 `WORK.md`
- `AGENTS.md`：给常见 coding agent 的根级约束文件
- `WORK.md`：当前项目唯一主控文件，也是事实来源
- `archive/`：值得保留的原始材料、结论快照、完成项目归档
- `scratch/`：临时脑暴、原始摘录、粗计划、一次性比对
- `src/`：实现代码
- `tests/`：验证代码或检查脚本

默认不要再创建额外文档。

### 目录治理

防止目录爆炸和文件乱放：

- **白名单目录**：上面 7 个目录（+ 根目录 3 个文件）就是全部。新建任何目录或根级文件前必须有明确理由，并在 WORK.md 材料表登记。
- **最大深度 3 层**：`src/auth/middleware/` 可以，`src/auth/middleware/helpers/utils/` 不行。如果需要第 4 层，先检查是否该合并或重组。
- **每个文件必须有归属**：
  - 代码 → `src/`
  - 测试 → `tests/`
  - 临时草稿 → `scratch/`（用完即删）
  - 值得保留的材料 → `archive/`
  - 项目主控信息 → `WORK.md`（不要拆成多个 markdown）
- **禁止的反模式**：
  - 不要建 `docs/`、`notes/`、`specs/`、`plans/` 等额外文档目录——所有规格和计划都在 WORK.md 里
  - 不要按阶段建目录（`phase-1/`、`step-2/`）——用 WORK.md 的状态流追踪阶段
  - 不要在根目录堆配置文件——项目配置放 `src/` 下，agent 配置放 `AGENTS.md`
- **接管已有仓库时**：尊重现有结构，不要强制重组。只补缺失的 `WORK.md` 和 `AGENTS.md`，把识别为临时性质的散落文件移入 `scratch/`。

## 何时使用

当用户需要下列任一能力时，使用这个 skill：
- 把一句话想法整理成可执行方案
- 把模糊需求收敛成问题卡、规格和任务
- 在 Git 仓库里建立轻量流程
- 用本地 worktree 并行推进任务
- 用最少文档承载调研、需求、计划、验证
- 保存中间材料，但避免越积越多
- 避免重复调研或重复决策
- 给 agent 提供稳定约束，但不引入复杂流程
- 不想走重型阶段文档，只想快速收口为可执行材料
- 只需要一份独立的问题卡或规格文档（轻量模式）

## 轻量模式

当用户明确只需要问题卡或规格（"帮我出一份问题卡""快速写个 spec"），不需要完整项目管理时，跳过仓库初始化和 WORK.md，直接输出独立文档。

### 问题卡结构

1. 问题是什么
2. 谁在什么场景下遇到
3. 现在怎么凑合做
4. 真正需要解决的需求
5. 已知约束
6. 当前假设
7. 关键决策点（难以后改的）
8. V1 包含 / 不包含
9. 成功标准

要求：问题描述不要写成技术实现；约束与假设分节；V1 In/Out 必须明确。

### 规格与任务结构

1. 一句话范围
2. 主要用户流
3. 关键组件与边界（≤ 9）
4. 选定方案
5. 风险与依赖
6. 验证策略
7. Sprint 切片
8. 任务清单（含 ID、标题、Depends On、Spec Ref、描述、Validation、Done Signal）

要求：Validation 必须具体到命令、检查动作或人工验收标准；Done Signal 必须可观察。

### 按需补充文档

- **执行编排**：任务数 > 5、存在并行边界、需要多 worktree 或 checkpoint 时补充。包含任务 DAG、并行波次、写入边界、Checkpoints、快测方案、停止/缩 scope 条件。
- **关键决策**：存在高代价、难以后改的决策时补充。包含决策主题、备选方案、取舍、最终决定、为什么现在必须定、哪些可以以后再定。

苏格拉底式交互在轻量模式下同样适用——仍然按交互原则提问和确认，只是产出格式不同。

## 初始化方式

当用户需要创建新仓库骨架时，运行：

```bash
python scripts/init_repo.py --root <repo-path> --title <项目标题> [--owner <负责人>]
```

该脚本会生成：README.md、AGENTS.md、WORK.md、archive/、scratch/、src/、tests/

主文件内容使用 `references/work-template.md`（WORK.md 的 15 个区块模板：Idea 含约束/假设/V1 边界、术语表、调研登记表、目标与非目标、范围图、架构与边界含模块 ≤9/接口/时序/限制、技术栈、实施计划含里程碑/任务拆解含 Validation+Done Signal/并行泳道/Deferred、风险、ADR-lite 含备选方案+结论、实施记录、验证表、状态报告、材料登记表、归档与删除记录）。
agent 约束内容使用 `references/agents-template.md`（AGENTS.md 短规则集：工作规则 14 条、方法论规则 7 条、去重规则 5 条、拆分规则 5 条、worktree 规则 4 条、归档规则 5 条）。

### 初始化后必须立即填充内容

运行 init_repo.py 只会生成空模板。**必须紧接着把用户的想法填入 WORK.md 的各个区块**，而不是留着空模板等以后再填。具体来说：

1. 把用户的一句话想法写入 1.1
2. 分析背景和痛点写入 1.2
3. 提炼成功信号写入 1.3
4. 分离约束（1.4）与假设（1.5），画 V1 边界（1.6）
5. 识别核心术语填入术语表
6. 区分目标和非目标
7. 梳理范围需求（≤ 9 组件）、架构模块、技术选型
8. 按 demoable sprint 拆解里程碑和任务，每个任务写清验证方式与完成信号
9. 识别风险，记录决策
10. 更新状态到 `frame` 或 `scope`

空模板是起点，不是终点。初始化完成时 WORK.md 应该已经是一份可执行的方案文档。

## 工作顺序

按六个阶段推进，每阶段有明确完成条件。前后阶段可以适度重叠，但不要跳过。

<HARD-GATE>
每个阶段的完成条件都是硬门——不得在用户确认前推进到下一阶段。即使任务看起来很简单，也要走完确认流程。确认可以很短（用户说"OK"即可），但不能跳过。
</HARD-GATE>

**反模式："这个太简单，直接做就行"**——每个项目都要走完交互流程。一个 todo list、一个单函数工具、一个配置变更——都一样。"简单"项目正是未检验假设最容易浪费工作的地方。

### 阶段 0：项目基建

1. 判断是初始化新仓库，还是接管已有仓库。
2. 把当前任务统一收敛到 `WORK.md`。
3. 生成或更新 `AGENTS.md`。接管已有仓库时，如果没有 `AGENTS.md` 则新建，已有则根据项目实际情况更新。
4. 把临时内容**实际移动**到 `scratch/`——不只是创建目录，还要用 `mv` 或等效操作把草稿、脑暴、原始笔记等文件移进去。

### 阶段 1：问题定锚

先确认正在解决的是"问题"，不是先入为主的"方案"。

- 明确：谁在什么场景下遇到问题、当前如何凑合、为什么现在值得做。
- 分开写：`Constraints`（客观限制）与 `Assumptions`（主观假设），不要混写。
- 先识别难以后改的关键决策（数据模型、权限模型、同步/异步模式等），再决定哪些必须现在拍板。
- 明确 V1 In/Out 边界——先画线，再填内容。
- 对非阻塞未知项做保守假设，不为小问题反复停下。
- 至少一条可验证的成功标准。

**交互检查点**：
- 区分两种情况：
  - **方案代替问题**（"用 React 重写 jQuery 后台"）：用户跳过了问题，直接给了方案。此时必须回溯到问题域——追问"现在的痛点具体在哪里？"
  - **明确意图，细节待补**（"做个 CLI 工具把 Markdown 转 PDF"）：用户的意图和场景已经清晰，不需要质疑意图本身。此时直接追问细节——谁在用、当前怎么凑合、V1 优先解决什么。
- V1 边界必须与用户确认——呈现"建议包含 / 建议不含 / 可选"三栏，让用户调整
- 如果识别出关键假设，询问假设是否成立，并说明假设成立与否对方案的影响

**阶段结束标准动作**：呈现问题定锚总结（一句话问题 + Constraints + Assumptions + V1 边界），同时审查：本阶段交互中产生的调研是否都已登记到 RES 表、涉及的决策是否已写入决策链（高代价的是否已升级到 ADR-lite）。等用户确认后再进入阶段 2。

**完成条件**：能用一句话描述问题（而非方案），Constraints/Assumptions 已分离，V1 边界已与用户确认。

### 阶段 2：规格收口

- 用一句话定义范围；若一句话都说不清，说明问题还没收口。
- 将方案压缩为 `≤ 9` 个主要组件/模块；超出说明范围过大或拆分不当。
- 只对高影响分歧做 `2~3` 个方案对比；不要为所有细节做比稿。
- 明确主要用户流、关键边界、依赖与验证策略。
- 保持规格薄：能指导实现即可，不要膨胀成长篇背景说明。
- 按"图表规范"中的触发条件，在 §6 架构与边界中绘制模块依赖图、数据流图、时序图。

**交互检查点**：
- 存在 2 条以上合理技术路径时，必须呈现 2~3 个方案（每个含逻辑链条、取舍、适用场景），标明推荐项，让用户选择
- 高代价决策（数据模型、同步/异步、外部绑定等）必须走 ADR-lite 流程：呈现备选方案 → 解释取舍 → 给出推荐 → 等用户拍板
- ADR-lite 必须保留被淘汰方案的关键论据：每个备选方案的核心优势、排除原因、相关调研证据（引用 RES-ID）。这样需求变化时可以快速重新评估，而不是从头调研。如果某个方案有实测数据或 PoC 结果，也要记录在对应 ADR 行中

**阶段结束标准动作**：呈现规格总结（一句话范围 + 模块列表 + 验证策略），同时审查：本阶段方案比选中的调研是否都已登记到 RES 表、被淘汰方案的论据是否已保留在 ADR-lite 中。等用户确认后再进入阶段 3。

**完成条件**：一句话范围可说清，组件 ≤ 9，验证策略已声明，高代价决策已由用户确认。

### 阶段 3：任务塑形

- 先按可演示（demoable）的 sprint 切片，而不是按技术层机械分组。
- 每个任务必须原子化、可独立提交、可独立 review。
- 每个任务必须写清 `Validation`（如何验证）与 `Done Signal`（何时算完）。
- 任务描述用动词开头，尽量短，不要夹带无关工作。
- 默认把"顺手做一下""后面再看"类内容放进 deferred，而不是混进当前任务。
- 任务过大的信号：标题含多个并列动作、跨多个不相关子系统、说不清"改了什么"、需要为依赖再拆分。

**交互检查点**：
- Sprint 切片方式如果有多种合理排法，呈现给用户选择（例如"先做核心链路再做边缘" vs "先做最高风险模块"）
- 如果发现范围比阶段 2 收口时更大，必须停下来向用户说明差异并确认是否扩展

**阶段结束标准动作**：呈现完整任务清单（含每个任务的 Validation + Done Signal），审查任务拆解过程中是否产生新的调研或决策需要沉淀。等用户 review 确认优先级和粒度后再进入阶段 4。

**完成条件**：每个任务有 Validation + Done Signal，无"顺手"任务，任务清单已由用户确认。

### 阶段 4：执行编排（按需）

仅在满足以下任一条件时才在 WORK.md 中补充执行编排信息：

- 任务数 `> 5`
- 明显存在可并行任务
- 需要多 worktree / 多 agent / 多 checkpoint
- 需要显式依赖图，否则执行顺序容易混乱

生成执行编排时：

- 为每个任务声明 `depends_on`、`owner_scope`、`validation_command`、`done_signal`。
- 按"图表规范"绘制任务 DAG 图，展示依赖和并行关系。
- 先排无依赖任务，再排集成与回归任务。
- 写清 checkpoint：已完成输出、验证结果、当前阻塞、下一步边界动作。
- 所有命令默认加 timeout；优先设计最短、最有证明力的快测。

**交互检查点**：
- 如果并行策略有多种合理编排（例如"先核心链路串行、再边缘并行" vs "按风险排序、高风险先跑"），呈现给用户选择
- 依赖图中如果有不确定的阻塞关系，必须确认后再定序

**阶段结束标准动作**：呈现执行编排总结（依赖图 + 并行泳道 + checkpoint 节点），等用户确认后再开始执行。

### 阶段 5：材料管理

- 把可复用结论回填进 `WORK.md`。
- 只有值得保留的原始材料才放进 `archive/`。
- 已吸收或已过期的临时材料及时删除。
- 执行调研去重检查（见下方规则）。

## 状态流

在 `WORK.md` 顶部维护状态：

`intake -> frame -> research -> scope -> design -> plan -> execute -> verify -> done`

## 主控文件规则

默认只维护一个 `WORK.md`。它必须短、表格化、可原地更新。

始终包含这些区块：
- 项目 UID 与当前状态
- idea（一句话想法、背景、成功信号、约束、假设、V1 边界）
- 术语表
- 调研登记表
- 目标与非目标
- 范围与功能映射
- 架构、职责边界、接口、时序
- 技术栈与方案概述
- 里程碑、任务（含验证方式与完成信号）、验证点
- 风险与应对
- ADR-lite
- 实施记录
- 当前状态报告
- 材料登记表
- 归档与删除记录

优先写短表，不写长文。

## 图表规范

表格适合列属性，但模块间关系、数据流向、时序分支、任务依赖用图才直观。在 WORK.md 的相关区块中按需绘制以下四种 ASCII 字符图。

### 何时画图

- 模块 ≥ 3 且存在调用/依赖关系 → 画模块依赖图
- 存在跨模块数据流或外部 IO → 画数据流图
- 交互涉及 ≥ 3 角色或存在分支/并发 → 画时序图
- 任务 ≥ 4 且存在依赖 → 画任务 DAG 图

模块 ≤ 2、交互角色 ≤ 2、任务 ≤ 3 且无并行时，表格和编号列表足够，不必强制画图。

### 1. 模块依赖图（§6.1 架构与边界）

用于展示模块之间的调用方向和依赖关系。放在 §6.1 模块表后面。

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ MOD-001  │────>│ MOD-002  │────>│ MOD-003  │
│ 列表选择态 │     │ 批量操作栏 │     │ 重试提交器 │
└──────────┘     └──────────┘     └──────────┘
                                       │
                                       v
                                 ┌──────────┐
                                 │ MOD-004  │
                                 │ 反馈展示  │
                                 └──────────┘
```

规则：
- 每个框标注 MOD-ID + 一句话职责
- 箭头 `──>` 表示调用/依赖方向
- 虚线 `- ->` 表示可选/弱依赖
- 外部系统用双线框 `╔══╗` 区分

### 2. 数据流图（§6.2-6.3 接口与时序）

用于展示数据在模块和外部系统之间的流转。放在 §6.2 接口表后面。

```
[用户勾选] ──> MOD-001 ──> MOD-002 ──> MOD-003 ──> IF-001(BE)
                                                      │
                                                      v
                                          MOD-004 <── [重试结果]
```

规则：
- `[名词]` 表示外部输入/输出
- 箭头上方可标注数据名（简短）
- 接口用 IF-ID 标注

### 3. 时序图（§6.3 交互时序）

当交互涉及 ≥ 3 角色或存在分支/并发时，替代编号列表。放在 §6.3。

```
运营          前端           后端
 │             │              │
 │──筛选失败──>│              │
 │──勾选多条──>│              │
 │──点击重试──>│              │
 │             │──POST /retry─>│ (循环 N 次)
 │             │<──200/409────│
 │<──汇总反馈──│              │
```

规则：
- 每列一个角色/系统，用竖线 `│` 作生命线
- 箭头 `──>` 从发起方指向接收方，上方标注动作
- 循环用括号标注 `(循环 N 次)`
- 分支用缩进 + 条件标注

### 4. 任务 DAG 图（§8.3 并行泳道 / 阶段 4 执行编排）

用于展示任务依赖和并行关系。放在 §8.3 或执行编排中。

```
TSK-001 ──> TSK-002 ──┬──> TSK-003 ──> TSK-004 ──┐
                       │                            ├──> TSK-006
                       └──> TSK-005 ──────────────┘
```

规则：
- 每个节点标注 TSK-ID
- `──>` 表示 depends_on
- 分叉 `┬` 和汇合 `┤` 表示并行与同步点
- 关键路径可用注释标注 `# 关键路径`

### 通用绘图规则

- 使用 Unicode box-drawing 字符（`┌ ┐ └ ┘ │ ─ ┬ ┤ ├ ┼ > v`），不用纯 ASCII 的 `+--+`
- 宽度控制在 80 字符以内，超宽则拆成多张
- 图的上方写一句话说明目的，下方可加简短注释
- 图和表格互补：图展示关系和流向，表格展示属性和细节
- 更新模块或任务时同步更新对应的图，保持一致性

## 命名规则

### 文件与目录命名

- **kebab-case**：所有文件名和目录名统一用小写 kebab-case（`my-feature-spec.md`，不用 camelCase 或 snake_case）
- **日期前缀**：需要时间排序的文件用 `YYYY-MM-DD-` 前缀（`2026-03-16-auth-redesign.md`）
- **语义后缀**：用途明确的文件加语义后缀区分类型：`-design.md`（设计）、`-plan.md`（计划）、`-notes.md`（笔记）
- **目录名 = 功能域**：目录名反映内容域而非阶段（用 `auth/`、`search/` 而不是 `phase-1/`、`step-2/`）
- **不用空格和特殊字符**：文件名只用 `a-z 0-9 -`，避免中文文件名（内容可以是中文）

### UID 规则

使用稳定编号，不要反复改号。

推荐模式：
- 项目：`PRJ-YYYYMMDD-001`
- 术语：`TERM-001`
- 调研：`RES-001`
- 目标：`G-001`
- 非目标：`NG-001`
- 范围项：`REQ-001`
- 模块：`MOD-001`
- 接口：`IF-001`
- 风险：`RSK-001`
- 决策：`ADR-001`
- 任务：`TSK-001`
- 验证：`VAL-001`
- 材料：`ART-001`

除非用户明确要求重构，否则不要重排已有编号。

## 调研去重规则

开始新调研前，必须先做这几步：

1. 查看 `WORK.md` 里的调研登记表。
2. 查看登记表链接到的 `archive/` 材料。
3. 判断旧结论是否仍然适用。
4. 只有在问题确实新、旧结论过期、或出现冲突时，才新增调研项。

每条调研项都要记录：问题、结论、证据路径/来源、上次检查时间、触发重查条件、复用说明。

常用重查条件：
- 时间敏感信息变化
- 需求边界变化
- 架构变化
- 旧证据质量过低
- 新旧结论冲突

### 去重原则

- 同一问题优先更新原 `RES` 行，不重复开新行。
- 同一来源优先复用旧摘要，不重复抄录。
- 同一结论若只补证据，补到原行即可。
- 有冲突时保留冲突说明，不要静默覆盖。
- 每次去重判定都要在**实施记录**中写明判定依据。

## 默认启发式

- 先问题，后方案。
- 最短闭环，而非最完整文档。
- **高影响决策必须询问用户，低影响细节做保守假设**——不要在琐碎问题上打断用户，也不要在关键分歧上替用户拍板。
- 询问时优先使用多选题格式，一次只问一个问题。每个选项写清逻辑链条和取舍，不要让用户在没有信息的情况下做决定。
- 规格必须薄，任务必须实。
- 每个任务都要可验证、可完成、可交付。
- 记录 deferred，不塞进当前任务。
- 快测优先，不默认全量回归。
- 不升级到重型阶段文档；保持轻量闭环。
- 输出 implementer 能直接执行的材料，而不是背景说明。

## 中间材料生命周期

只用三层：

### 1. WORK.md
存放可复用结论、边界、计划、决策、状态。

### 2. scratch/
存放临时材料：脑暴草稿、原始摘录、粗计划、一次性比较、临时命令记录。

规则：允许快速记、允许粗糙、及时吸收、吸收后删除、过期即删除。接管已有仓库时，把识别为临时性质的文件**实际移动**到 `scratch/`。

### 3. archive/
只存值得保留的材料。仅当满足至少一项时才归档：未来可能复查、重建成本高、支撑关键决策、未来大概率复用、需要审计或追溯、已完成项目需要留痕。

不要归档：已在 `WORK.md` 充分吸收、完全重复、低价值脑暴噪音、已失效且不可复用。

把保留材料登记到 `WORK.md` 的材料表。

## 拆分规则

默认不拆文件。仅当满足任一条件时才拆：表太长主文件难扫读、原始证据过大、子主题有独立生命周期、某决策需要完整 ADR、用户明确要求分离。

拆分后必须满足：`WORK.md` 仍是总入口、子文件顶部写回链、子文件继承项目 UID、写明拆分原因。

## Git 与 worktree 规则

当代码改动较大，或需要并行推进时，使用一任务一 worktree。

默认模式：
- 主分支保持干净
- 一任务一分支
- 一任务一 worktree
- 不混做无关任务
- 验证后尽快合并
- 合并后及时清理 worktree

推荐 worktree 路径：`../<project-name>-<task-uid>`

推荐分支名：`feat/<project-uid>-<task-uid>`、`fix/<project-uid>-<task-uid>`、`docs/<project-uid>-<task-uid>`

## 提交规则

小步提交，尽早提交，频繁提交。

提交信息建议：

```text
feat(scope): add minimal api contract
fix(core): handle empty input in parser
docs(work): clarify non-goals and verification points
refactor(plan): split validator and runner
```

如果当前提交只影响文档或流程，scope 可写成：
- `work`
- `agent`
- `research`
- `plan`
- `verify`

## AGENTS.md 兼容性说明

对代码仓库本身：
- `AGENTS.md` 是更通用的 agent 约束入口
- 常见 coding agent 更容易自动读取根级 `AGENTS.md`
- 因此默认在仓库根目录生成 `AGENTS.md`

## 输出要求

默认输出应尽量满足：
- 文档最少
- 单文件可推进
- 规则短句化
- 表格优先
- UID 稳定
- 状态持续更新
- 中间材料可追踪
- 可归档、可删除、可复用
- 便于 agent 接管
- 每个任务可验证、可交付

## 何时读取附加资料

- 读取 `references/work-template.md` 获取 WORK.md 的标准模板与字段要求。模板包含 15 个区块：Idea（一句话想法、背景、成功信号、约束、假设、V1 边界）、术语表、调研登记表（含重查条件和复用说明列）、目标与非目标、范围图（含一句话范围）、架构与边界（模块 ≤9、接口、交互时序、关键限制）、技术栈与方案概述、实施计划（里程碑按 demoable sprint 切片、任务拆解含 depends_on/Validation/Done Signal、并行泳道、Deferred 延期项）、风险与应对、ADR-lite（含备选方案和结论列）、实施记录、验证表、当前状态报告、材料登记表、归档与删除记录。
- 读取 `references/agents-template.md` 获取 AGENTS.md 的标准模板。模板是短规则集，包含：工作规则 14 条（单文件优先、先问题后方案、先查旧调研等）、方法论规则 7 条（Constraints/Assumptions 分开写、V1 In/Out 明确、组件 ≤9、按 demoable sprint 切片等）、去重规则 5 条、拆分规则 5 条、worktree 规则 4 条、归档规则 5 条。
- 读取 `references/decision-rules.md` 获取问题定锚、关键决策、任务粒度和执行编排的判定规则。包含 6 个判定规则：①问题定锚（6 条必满足条件）、②关键决策判定（6 类高代价 vs 4 类低代价）、③规格收口（6 条合格标准）、④任务粒度（5 条单任务标准 + 4 个过大信号）、⑤执行编排触发条件（5 条任一触发）、⑥默认执行启发式（5 条优先原则：最短闭环、少量高价值问题、记录延期项、快测优先、输出可执行材料）。
- 读取 `references/example-project.md` 获取一份完整中文示例（"失败任务批量重试"），展示从一句话用户请求到填充完成的 WORK.md 全 15 个区块、决策链记录 4 轮交互示例、以及轻量模式使用时机说明。
- 需要快速起目录与模板文件时，运行 `scripts/init_repo.py`。脚本接受 `--root`（仓库路径）、`--title`（项目标题）、`--owner`（负责人）参数，生成 7 件套：README.md、AGENTS.md、WORK.md、archive/、scratch/、src/、tests/。
