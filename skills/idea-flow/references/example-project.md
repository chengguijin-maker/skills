# 中文示例：为失败任务增加批量重试能力

## 1. 原始用户请求

> 我们后台任务列表里经常有失败任务，运营需要一个"批量重试"能力。帮我用 idea-flow 整理一下。

## 1. Idea

### 1.1 一句话想法
运营能在任务列表中批量选择失败任务并一次性发起重试。

### 1.2 背景
运营同学在后台任务列表中处理失败记录时，只能逐条点击"重试"，或导出后人工记录再处理。每天有几十条失败任务，重复操作成本高且容易漏掉。

### 1.3 成功信号
- 运营可在 1 次操作中提交多条失败任务重试
- 批量操作明显减少逐条点击次数
- 提交后有明确成功/失败反馈

### 1.4 约束（Constraints）

客观限制，不可协商：

- 当前列表接口已支持失败状态筛选
- 重试接口目前只支持单条调用
- 本期不改任务执行引擎内部逻辑

### 1.5 假设（Assumptions）

主观假设，可能需要验证：

- 单次批量上限 50 条足够覆盖主要场景
- 运营只需要看到"已提交重试"反馈，不要求实时逐条结果流式刷新

### 1.6 V1 边界

| 类型 | 内容 |
|---|---|
| V1 包含 | 列表多选、批量重试入口、基础结果反馈、失败保护 |
| V1 不含 | 重试历史面板、逐条进度实时推送、失败原因聚合分析 |
| 延期项 | 重试历史面板（V2 考虑）、失败原因聚合分析（V2 考虑） |

### 6.3 交互时序

1. 运营筛选失败任务
2. 勾选多条记录
3. 点击"批量重试"
4. 前端弹出确认对话框
5. 确认后逐条调用 IF-001
6. 汇总结果展示成功/失败数

### 6.4 关键限制

- 单次批量上限 50 条
- 前端循环调用，无后端批量接口

## 4. 何时用轻量模式

如果用户的请求是"帮我出一份问题卡"或"快速写个 spec"，不需要完整的项目管理流程，则跳过仓库初始化，直接按以下结构输出独立文档：

- **问题卡**：问题是什么 → 谁遇到 → 现在怎么凑合 → 真正需求 → 约束 → 假设 → 关键决策点 → V1 边界 → 成功标准
- **规格与任务**：一句话范围 → 用户流 → 关键组件 ≤9 → 选定方案 → 风险依赖 → 验证策略 → sprint 切片 → 任务清单（含 Validation + Done Signal）

完整项目管理才走 WORK.md 路径。
