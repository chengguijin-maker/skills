# Idea Ops Lite（简体中文）

## 目标

把零散 idea 迅速收敛成一个**可执行、可验证、可迭代**的最小工作流。

默认采用**单文件优先**模式：
- 仓库根目录只保留一个主控文件 `WORK.md`
- 仓库根目录使用 `AGENTS.md` 作为 agent 约束入口
- 只有在内容过长、证据过重、生命周期独立时才拆文件

## 默认仓库结构

优先生成或维护这个最小结构：

```text
repo/
├── README.md
├── AGENTS.md
├── WORK.md
├── archive/
├── scratch/
├── src/
└── tests/
```

含义：
- `README.md`：一句话入口，指向 `WORK.md`
- `AGENTS.md`：给常见 coding agent 的根级约束文件
- `WORK.md`：当前项目唯一主控文件，也是事实来源
- `archive/`：值得保留的原始材料、结论快照、完成项目归档
- `scratch/`：临时脑暴、原始摘录、粗计划、一次性比对
- `src/`：实现代码
- `tests/`：验证代码或检查脚本

默认不要再创建额外文档。

## 何时使用

当用户需要下列任一能力时，使用这个 skill：
- 把一句话想法整理成可执行方案
- 在 Git 仓库里建立轻量流程
- 用本地 worktree 并行推进任务
- 用最少文档承载调研、需求、计划、验证
- 保存中间材料，但避免越积越多
- 避免重复调研或重复决策
- 给 agent 提供稳定约束，但不引入复杂流程

## 初始化方式

当用户需要创建新仓库骨架时，运行：

```bash
python scripts/init_repo.py --root <repo-path> --title <项目标题> [--owner <负责人>]
```

该脚本会生成：
- `README.md`
- `AGENTS.md`
- `WORK.md`
- `archive/`
- `scratch/`
- `src/`
- `tests/`

主文件内容使用 `references/work-template.md`。
agent 约束内容使用 `references/agents-template.md`。

### 初始化后必须立即填充内容

运行 init_repo.py 只会生成空模板。**必须紧接着把用户的想法填入 WORK.md 的各个区块**，而不是留着空模板等以后再填。具体来说：

1. 把用户的一句话想法写入 1.1
2. 分析背景和痛点写入 1.2
3. 提炼成功信号写入 1.3
4. 识别核心术语填入术语表
5. 区分目标和非目标
6. 梳理范围需求、架构模块、技术选型
7. 拆解里程碑和任务
8. 识别风险，记录决策
9. 更新状态到 `frame` 或 `scope`

空模板是起点，不是终点。初始化完成时 WORK.md 应该已经是一份可执行的方案文档。

## 工作顺序

按这个顺序推进：

1. 判断是初始化新仓库，还是接管已有仓库。
2. 把当前任务统一收敛到 `WORK.md`。
3. 生成或更新 `AGENTS.md`。无论是初始化还是接管，都必须确保仓库根目录有 `AGENTS.md`。接管已有仓库时，如果没有 `AGENTS.md` 则新建，已有则根据项目实际情况更新。
4. 把临时内容**实际移动**到 `scratch/`——不只是创建目录，还要用 `mv` 或等效操作把草稿、脑暴、原始笔记等文件移进去。
5. 把可复用结论回填进 `WORK.md`。
6. 只有值得保留的原始材料才放进 `archive/`。
7. 已吸收或已过期的临时材料及时删除。

## 状态流

在 `WORK.md` 顶部维护状态：

`intake -> frame -> research -> scope -> design -> plan -> execute -> verify -> done`

## 主控文件规则

默认只维护一个 `WORK.md`。它必须短、表格化、可原地更新。

始终包含这些区块：
- 项目 UID 与当前状态
- idea 与术语表
- 调研登记表
- 目标与非目标
- 范围与功能映射
- 架构、职责边界、接口、时序
- 技术栈与方案概述
- 里程碑、任务、验证点
- 风险与应对
- ADR-lite
- 实施记录
- 当前状态报告
- 材料登记表
- 归档与删除记录

优先写短表，不写长文。

## UID 规则

使用稳定编号，不要反复改号。

推荐模式：
- 项目：`PRJ-YYYYMMDD-001`
- 术语：`TERM-001`
- 调研：`RES-001`
- 目标：`G-001`
- 非目标：`NG-001`
- 范围项：`REQ-001`
- 模块：`MOD-001`
- 接口：`IF-001`
- 风险：`RSK-001`
- 决策：`ADR-001`
- 任务：`TSK-001`
- 验证：`VAL-001`
- 材料：`ART-001`

除非用户明确要求重构，否则不要重排已有编号。

## 调研去重规则

开始新调研前，必须先做这几步：

1. 查看 `WORK.md` 里的调研登记表。
2. 查看登记表链接到的 `archive/` 材料。
3. 判断旧结论是否仍然适用。
4. 只有在问题确实新、旧结论过期、或出现冲突时，才新增调研项。

每条调研项都要记录：
- 问题
- 结论
- 证据路径或来源
- 上次检查时间
- 触发重查条件
- 复用说明

常用重查条件：
- 时间敏感信息变化
- 需求边界变化
- 架构变化
- 旧证据质量过低
- 新旧结论冲突

### 去重原则

- 同一问题优先更新原 `RES` 行，不重复开新行。
- 同一来源优先复用旧摘要，不重复抄录。
- 同一结论若只补证据，补到原行即可。
- 有冲突时保留冲突说明，不要静默覆盖。
- 每次去重判定都要在**实施记录**中写明判定依据（例如"RES-002 与 RES-001 去重判定：问题层次不同，RES-001 是选型，RES-002 是性能验证"），这样后续接手的人能理解为什么新增而非复用。

## 中间材料生命周期

只用三层：

### 1. WORK.md
存放可复用结论、边界、计划、决策、状态。

### 2. scratch/
存放临时材料：
- 脑暴草稿
- 原始摘录
- 粗计划
- 一次性比较
- 临时命令记录

规则：
- 允许快速记
- 允许粗糙
- 及时吸收
- 吸收后删除
- 过期即删除
- 接管已有仓库时，把识别为临时性质的文件（草稿笔记、脑暴记录、粗计划等）**实际移动**到 `scratch/`，而不是仅在材料登记表中标注分类

### 3. archive/
只存值得保留的材料。

仅当满足至少一项时才归档：
- 未来可能复查
- 重建成本高
- 支撑关键决策
- 未来大概率复用
- 需要审计或追溯
- 已完成项目需要留痕

不要归档这些内容：
- 已在 `WORK.md` 充分吸收
- 完全重复
- 低价值脑暴噪音
- 已失效且不可复用

把保留材料登记到 `WORK.md` 的材料表。

## 拆分规则

默认不拆文件。

仅当满足任一条件时才拆：
- 表太长，主文件难扫读
- 原始证据过大
- 子主题有独立生命周期
- 某决策需要完整 ADR
- 用户明确要求分离

拆分后必须满足：
- `WORK.md` 仍是总入口
- 子文件顶部写回链
- 子文件继承项目 UID
- 写明拆分原因

## Git 与 worktree 规则

当代码改动较大，或需要并行推进时，使用一任务一 worktree。

默认模式：
- 主分支保持干净
- 一任务一分支
- 一任务一 worktree
- 不混做无关任务
- 验证后尽快合并
- 合并后及时清理 worktree

推荐 worktree 路径：

```text
../wt/<project-uid>/<task-uid>
```

推荐分支名：

```text
feat/<project-uid>-<task-uid>
fix/<project-uid>-<task-uid>
docs/<project-uid>-<task-uid>
```

## 提交规则

小步提交，尽早提交，频繁提交。

提交信息建议：

```text
feat(scope): add minimal api contract
fix(core): handle empty input in parser
docs(work): clarify non-goals and verification points
refactor(plan): split validator and runner
```

如果当前提交只影响文档或流程，范围可写成：
- `work`
- `agent`
- `research`
- `plan`
- `verify`

## AGENTS.md 兼容性说明

对 skill 包本身：
- `agents/openai.yaml` 是 ChatGPT skill 所需元数据
- `AGENTS.md` 不是 skill 打包必需项

对代码仓库本身：
- `AGENTS.md` 是更通用的 agent 约束入口
- 常见 coding agent 更容易自动读取根级 `AGENTS.md`
- 因此默认在仓库根目录生成 `AGENTS.md`
- 不再默认使用 `.agent/steering.md`

## 输出要求

默认输出应尽量满足：
- 文档最少
- 单文件可推进
- 规则短句化
- 表格优先
- UID 稳定
- 状态持续更新
- 中间材料可追踪
- 可归档、可删除、可复用
- 便于 agent 接管
