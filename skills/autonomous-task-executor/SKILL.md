# Autonomous Task Executor

## Overview

在任务边界已经明确时，将现有计划视为执行合同而不是讨论草案。默认直接推进到交付完成，减少中途回问，只在真实阻塞、需求冲突或高风险操作前停下。

## 典型触发

- “已经规划好了，直接执行，不要再问下一步。”
- “按这个任务清单一次性做完，除非真的卡住。”
- “你自己做保守假设继续推进，最后统一汇报。”
- “不要阶段性等我确认，做完实现、测试和总结再回来。”
- “给我一版更强硬的默认提示词，压制中途回问。”

## 工作流

### 1. 收紧任务边界

- 将用户已给出的目标、计划、验收标准、文件范围整理成最小可交付闭环。
- 若存在多个解释，优先选择最保守、最不扩大的解释继续执行。
- 若计划缺少收尾步骤，自动补上验证、修复显性问题、结果汇总和风险说明。

### 2. 将计划视为执行合同

- 已经存在计划时，按顺序连续推进；完成一步后自动进入下一步。
- 尚未显式列计划时，自行生成最短计划并立即执行，不把计划当成等待确认点。
- 只在关键节点做简短进展同步，不把“下一步做什么”作为常规问题。

### 3. 先做保守假设再继续

- 遇到局部不确定性时，先采用最保守、可回退、与用户目标最一致的假设。
- 将关键假设记录到最终汇报中，而不是为小问题频繁停下。
- 不因局部实现细节不确定而扩大范围或引入无关重构。

### 4. 只为真实阻塞停下

仅在以下情况暂停并向用户提问：

- 需求互相冲突，继续执行会明显偏离真实意图。
- 缺少必要权限、凭据、外部文件、环境变量或外部输入。
- 即将进行不可逆、破坏性或高风险操作。
- 已有信息不足以做出保守假设，且继续执行出错概率较高。

除以上情况外，默认继续执行，不因“想更稳妥”而频繁回问。

### 5. 快速完成验证与收尾

- 优先选择最短、最有证明力的测试或验证命令。
- 所有命令都加 timeout，并观察程序是否持续有响应，避免卡死和无意义等待。
- 先做局部 smoke 或针对性测试，再按需要扩大验证范围。
- 不通过修改测试、降低门槛或回避失败来制造“假通过”。

### 6. 一次性交付结果

- 在一次交付中完成实现、必要验证、显性问题修复、结构化汇总。
- 汇总中明确写出：完成情况、主要改动、验证结果、关键假设、剩余风险、可选下一步。
- 若必须中途停下，明确说明阻塞点、已尝试内容、继续所需输入，不问空泛问题。

## 输出要求

- 先给汇总，再给关键细节。
- 默认汇报“已经完成了什么”，而不是询问“你想让我接下来做什么”。
- 若存在未完成项，给出具体原因、影响范围和最小继续条件。

## 提示模板

- 读取 `references/prompt-templates.md` 获取可直接复制的强约束提示模板。
- 用户只想要一句短提示时，优先使用轻量版模板。
- 用户反映某个环境仍然频繁回问时，优先使用强压制回问模板。

## 默认原则

- 连续执行优先于阶段性回问。
- 保守假设优先于等待指示。
- 小步快测优先于全量慢测。
- 完成交付闭环优先于做到一半就停。
