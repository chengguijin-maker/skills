## 当前问题

你们团队十几个人，文档分散在四个平台上：

- **飞书** — 可能是日常沟通、会议纪要、部分项目文档
- **Notion** — 可能是产品文档、知识沉淀、Wiki 类内容
- **Google Docs** — 可能是协作文档、对外分享的内容
- **Git 仓库** — 技术文档、README、设计文档、ADR 等

核心痛点是 **"找东西很痛苦"**，这背后通常包含几个具体问题：

1. **不知道去哪找** — 同一类文档可能存在多个平台
2. **搜索割裂** — 每个平台只能搜自己的内容，没有统一搜索
3. **信息过时** — 多处存放导致版本不一致，不知道哪个是最新的
4. **缺乏结构** — 没有统一的分类体系和命名规范

## 做统一知识库之前需要想清楚的几个问题

在选方案之前，建议先回答这些问题，它们会直接影响技术选型和落地路径：

### 1. 目标是"统一入口"还是"统一平台"？

- **统一入口（轻量方案）**：保留现有平台，建一个聚合搜索/导航层，降低查找成本
- **统一平台（重方案）**：把所有文档迁移到一个平台上，从根本上解决分散问题

两种路径的成本和收益差别很大。

### 2. 文档的主要类型有哪些？

不同类型的文档适合不同的管理方式：

| 文档类型 | 特点 | 举例 |
|---------|------|------|
| 过程类 | 时效性强，写完很少改 | 会议纪要、周报、复盘 |
| 知识类 | 需要持续维护更新 | 技术方案、API 文档、操作手册 |
| 协作类 | 多人实时编辑 | 需求文档、设计稿评审 |
| 代码类 | 和代码仓库强绑定 | README、CHANGELOG、架构决策 |

### 3. 团队的技术水平和习惯是什么样的？

这决定了方案的可落地性：

- 团队成员是否都能接受 Markdown？
- 是否有人维护知识库（兼职 or 专职）？
- 大家对"多一个工具"的接受度如何？

### 4. 预算和维护投入？

- 是否愿意付费使用 SaaS 产品？
- 是否有能力自建和运维？
- 能投入多少人力做初始迁移和持续维护？

## 常见的方案方向

根据上面的回答，大致有这几条路：

**方向 A：选一个主平台，逐步收敛**
> 比如全部收到 Notion / 飞书文档 / Confluence 上，制定规范，逐步迁移。适合团队规模不大、文档量可控的情况。

**方向 B：搭建聚合搜索层**
> 保留各平台，用工具（如 Glean、Dashworks，或自建方案）做统一搜索和索引。适合迁移成本太高、平台各有不可替代优势的情况。

**方向 C：自建知识库系统**
> 用开源方案（如 Outline、BookStack、Wiki.js）或自研，把核心文档迁移过来。适合有技术能力、对数据掌控有要求的团队。

**方向 D：Git 为核心 + 静态站点**
> 文档全部 Markdown 化，存 Git 仓库，用 Docusaurus / VitePress / MkDocs 生成站点。适合技术团队、文档以知识类为主的场景。

## 建议的下一步

1. **先盘点现状** — 各平台分别有多少文档、什么类型、使用频率如何
2. **明确上面四个问题的答案** — 确定约束条件
3. **小范围试点** — 选一个方向，先迁移一个模块（比如新人入职文档），验证流程是否跑得通
4. **再推广** — 根据试点经验调整方案，逐步铺开
