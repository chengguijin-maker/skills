# 判定规则与默认启发式

## 1. 问题定锚规则

满足以下全部条件，才算问题已经定锚：

- 能用一句话说明问题本身，而不是实现方案。
- 明确至少一个真实用户 / 场景。
- 写清当前替代做法或现有痛点。
- `Constraints` 与 `Assumptions` 已分离。
- `V1 In / Out` 已明确。
- 有至少一条可验证成功标准。

若不满足，继续补 `problem-frame.md`，不要急着写规格。

## 2. 关键决策判定

以下类型通常属于高代价决策，需要写入 `critical-decisions.md`：

- 数据真相源或主存储模型
- 身份 / 权限模型
- 同步 / 异步 / 轮询 / 回调模式
- 外部服务或供应商绑定
- 高影响 UI 交互范式
- 明确影响兼容性、迁移成本或并发模型的决定

以下内容通常不需要单独建档：

- 文案措辞
- 非关键页面微调
- 易于回退的局部实现细节
- 当前 sprint 内即可验证并低成本改动的选项

## 3. 规格收口规则

一个合格的 `spec-and-tasks.md` 应满足：

- 一句话范围清楚
- 关键组件 / 边界 `<=9`
- 只对高影响分歧做方案对比
- 已写明依赖与风险
- 已给出验证策略
- 任务清单能直接进入执行，不需要再补关键设计决定

## 4. 任务粒度规则

单个任务应满足：

- 单一目标
- 一次提交可完成或接近一次提交可完成
- 不跨多个不相关子系统
- 有明确 `Validation`
- 有明确 `Done Signal`

以下是任务过大的信号：

- 标题里包含多个并列动作
- 描述涉及多个模块的大范围改动
- 无法用一句话说清“做完后能看到什么变化”
- 需要拆开后才能安排依赖

## 5. 执行编排触发条件

满足任一条件时，生成 `execution-plan.md`：

- 任务数 `> 5`
- 存在明显并行边界
- 涉及多 agent / 多 worktree
- 需要阶段性 checkpoint
- 不画依赖图就容易乱序或冲突

否则直接按 `spec-and-tasks.md` 推进即可，不额外建编排文档。

## 6. 默认执行启发式

- 优先最短闭环，而不是最完整闭环。
- 优先少量高价值问题，不做长串低价值澄清。
- 优先记录延期项，不把非关键项塞进当前 V1。
- 优先快测与局部验证，不默认全量回归。
- 优先输出 implementer 能直接执行的材料，而不是背景说明。
